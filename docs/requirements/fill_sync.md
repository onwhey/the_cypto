# 011 FillSync Requirements

> 建议路径：`docs/requirements/fill_sync.md`  
> 阶段：011  
> 模块：FillSync / 成交明细同步  
> 上游：010 OrderStatusSync  
> 下游：无直接订单下游；012 Account Overview / One-Click Sync 为独立账户总览模块  
> 状态：需求定稿草案  
> 核心结论：011 只查询并保存成交明细与订单成交汇总；不做对账、不更新持仓、不释放订单锁。

---

## 1. 阶段目的

011 只负责一件事：

```text
基于 010 已经确认可查询的订单，
重新向 Binance 查询该订单对应的成交明细，
逐条保存 Binance 返回的成交记录，
并生成以 009 提交记录为主线的一条订单成交汇总。
```

011 不依赖 009 下单接口返回作为最终成交依据。

011 不依赖 010 的订单状态字段推导成交结果。

011 的成交事实必须来自 011 自己重新请求 Binance 成交明细接口得到的数据。

---

## 2. 当前业务前提

当前 009 阶段提交的是：

```text
市价单
```

一笔 009 市价单可能被 Binance 撮合成：

```text
1 笔成交
多笔成交
0 笔成交或暂时查不到成交
```

多笔成交是正常现象。

示例：

```text
009 提交 1 笔市价买单。
Binance 实际撮合成 3 笔成交：
  第 1 笔：价格 A，数量 X
  第 2 笔：价格 B，数量 Y
  第 3 笔：价格 C，数量 Z
```

011 必须保存这 3 条逐笔成交明细，同时生成一条该 009 订单的成交汇总。

---

## 3. 010 与 011 必须分开

010 查询订单状态。

011 查询成交明细。

两者使用不同 Binance 接口。

010 使用：

```text
USDS-M：GET /fapi/v1/order
COIN-M：GET /dapi/v1/order
```

011 使用：

```text
USDS-M：GET /fapi/v1/userTrades
COIN-M：GET /dapi/v1/userTrades
```

因此：

```text
010 只问：这笔订单 Binance 是否认识？当前订单状态是什么？
011 才问：这笔订单具体成交了几笔？每笔价格、数量、手续费是多少？
```

011 不得重新实现订单状态查询。

010 不得实现成交明细同步。

---

## 4. 009 提交记录主线与外键要求

011 的所有成交相关记录都必须以 009 提交记录作为主线关联对象。

业务要求：

```text
1. 009 提交记录是订单提交、状态查询、成交同步、对账链路的主入口。
2. 011 生成的每一条成交明细记录，都必须能直接关联回对应的 009 提交记录。
3. 011 生成的订单成交汇总，也必须能直接关联回对应的 009 提交记录。
4. 未来 UI 打开某一条 009 提交记录详情时，必须能展开查看该订单对应的：
   - 010 订单状态查询结果；
   - 011 逐笔成交明细；
   - 011 订单成交汇总。
5. 011 不得只按 Binance orderId、客户订单编号、symbol 或时间范围孤立保存成交结果。
```

关系要求：

```text
一条 009 提交记录可以对应 0 条、1 条或多条 011 成交明细记录。

一条 009 提交记录最多对应 1 条 011 订单成交汇总。
```

含义：

```text
0 条成交明细：
  011 尚未执行，或查询成功但 Binance 返回 0 条成交，或查询失败。

1 条成交明细：
  该订单被 Binance 一次撮合成交。

多条成交明细：
  该订单被 Binance 拆成多笔撮合成交。
```

最终 UI / 审计主线应为：

```text
009 提交记录
  ├── 010 订单状态查询记录
  ├── 011 成交明细记录
  └── 011 订单成交汇总
```

012 Account Overview / One-Click Sync 是独立账户总览模块，不挂在 009 订单详情主线下，不作为订单对账结果。

这条是 P0 追溯要求。否则后续 UI、审计、排错会依赖 symbol、orderId、时间等字段拼接数据，风险较高。

---

## 5. 011 的输入范围

011 的默认输入来自 010。

只有当 010 已经成功保存该订单的状态查询证据后，011 才进入成交明细同步。

P0 默认只处理：

```text
本轮 010 已成功同步订单状态的那一笔订单。
```

011 不得自行查询数据库历史最新订单。

011 不得在 010 未执行、010 unknown、010 not_found、010 blocked、010 failed 的情况下，自动绕过 010 直接查询成交。

如果需要人工排查，后续可以提供显式手工入口，但不得作为 P0 默认流程。

---

## 6. 本轮无 010 成功订单时的处理

如果本轮没有 010 成功同步的订单，011 必须跳过。

要求：

```text
1. 不请求 Binance。
2. 不回查历史订单。
3. 不生成成交明细记录。
4. 不生成订单成交汇总。
5. 只记录普通运行日志，说明本轮没有可同步成交的订单。
6. 不写风险告警，除非上游状态本身是异常。
```

示例：

```text
8:00 策略决定不操作。
009 没有提交订单。
010 跳过。
011 也必须跳过，不请求 Binance。
```

## 7. Binance 查询接口

### 7.1 USDS-M Futures

011 查询 U 本位成交明细使用：

```text
GET /fapi/v1/userTrades
```

查询参数原则：

```text
symbol 必须明确。
orderId 应优先传入。
timestamp 必须有。
```

### 7.2 COIN-M Futures

011 查询币本位成交明细使用：

```text
GET /dapi/v1/userTrades
```

查询参数原则：

```text
symbol 必须明确。
orderId 应优先传入。
timestamp 必须有。
```

COIN-M 不得把 contracts 转成 base asset 或 quote asset 后再查询。

## 8. 查询编号规则

011 查询成交明细时，优先使用：

```text
Binance 订单编号。
```

原因：

```text
Binance 成交明细接口支持按 orderId 查询成交记录；
orderId 应来自 010 查询订单状态时 Binance 返回的订单信息，或 009 提交成功时保存的交易所订单号。
```

如果没有 Binance 订单编号：

```text
011 不请求 Binance；
记录 failed_before_query；
写明缺少可查询订单编号；
写审计/告警。
```

P0 不建议只凭时间窗口和交易对宽泛查询成交，因为这样容易把其他订单的成交混进来。

## 9. 市场域规则

011 必须支持：

```text
USDS-M Futures / fapi
COIN-M Futures / dapi
```

但运行时只能激活一个市场域。

要求：

```text
1. 当前激活 U 本位时，只能查询 U 本位成交明细。
2. 当前激活币本位时，只能查询币本位成交明细。
3. 订单所属市场域与当前激活市场域不一致时，不得请求 Binance。
4. 不一致时必须记录阻断原因，并写审计/告警。
```

011 不能把 U 本位和币本位并行查询。

## 11. 订单成交汇总

011 除了保存逐笔成交明细，还必须生成一条面向 009 提交记录的订单成交汇总。

这条汇总是未来 UI 查看“每次 009 提交的这一笔总订单实际成交结果”的主要入口。

汇总必须直接关联：

```text
对应的 009 提交记录
对应的 010 订单状态查询记录
```

关系要求：

```text
一条 009 提交记录最多对应一条订单成交汇总。
```

成交汇总至少应覆盖以下信息方向：

```text
对应的 009 提交记录
对应的 010 订单状态查询记录
交易所
市场类型
账户域
运行时 active 市场域
endpoint family

Binance 订单编号
本系统客户订单编号

总成交 quote 金额或 base 金额
成交均价
总手续费

第一笔成交时间
```

注意：

```text

```

011 不得基于该汇总更新持仓、释放订单锁或判断交易完成。

## 12. 汇总计算口径

011 可以基于本次同步到的逐笔成交明细生成汇总。

建议口径：

```text
总成交数量：
  所有成交明细数量求和。

总成交金额：
  U 本位优先使用 quote 数量求和；
  币本位按 Binance 返回字段保持原始口径，不得擅自把 contracts 转成 base asset 或 quote asset。

成交均价：
  使用总成交金额 / 总成交数量。
  如果字段口径不足以安全计算，则可以为空，并在 reason_code 中说明。

总手续费：
  按手续费资产分组汇总。
  如果一笔订单存在多个手续费资产，必须保留资产维度，不得强行合并为一个资产。

realized pnl：
  可按 Binance 返回字段汇总保存。
  但不得据此更新账户、持仓或绩效报表。
```

如果查询返回 0 条成交：

```text
成交明细条数 = 0
成交同步结果 = synced_empty
总成交数量 = 0 或空值
成交均价 = 空值
```

不得据此直接判断订单没有成交或风险解除。

## 13. 去重与幂等

011 必须支持幂等。

同一笔 Binance 成交明细不得重复入库。

建议唯一性口径：

```text
exchange
market_type
account_domain
symbol
Binance orderId
Binance trade id
```

如果同一成交明细已经存在：

```text
1. 不重复插入。
2. 可以记录 skipped_duplicate 或更新审计时间。
3. 不改变原始成交事实。
4. 订单成交汇总不得因重复运行而重复累计。
```

重复运行 011 不应造成：

```text
重复手续费
重复成交数量
重复成交金额
重复 realized pnl
```

## 14. 查询结果分类

011 本地同步结果建议分为：

```text
synced
synced_empty
unknown
failed_before_query
blocked_before_query
skipped_no_order_status
recovery_skipped_out_of_window
```

含义：

```text
synced：
  成功请求 Binance，并同步到一条或多条成交明细，同时生成或更新订单成交汇总。

synced_empty：
  成功请求 Binance，但 Binance 返回 0 条成交明细。
  对当前市价单来说这应视为需要关注的结果，但 011 仍只保存事实，不做交易结论。

unknown：
  本次查询没有得到可信结论。
  例如网络错误、超时、Binance 5xx、限频、连接异常。

failed_before_query：
  查询前本地条件不满足。
  例如缺少 Binance 订单编号、缺少 symbol、缺少上游 010 证据。

blocked_before_query：
  查询前被系统规则阻断。
  例如市场域不一致。

skipped_no_order_status：
  本轮没有 010 成功订单状态证据，011 正常跳过。

recovery_skipped_out_of_window：
  异常恢复时发现漏同步成交，但已经超过自动恢复窗口，011 不自动请求 Binance。
```

## 15. synced_empty 的处理

如果 Binance 成交明细接口请求成功，但返回 0 条成交：

```text
本地结果 = synced_empty
```

011 不得直接推导：

```text
订单没有成交
可以重新下单
可以释放订单锁
可以更新持仓为 0
```

对于当前市价单来说，0 条成交需要人工确认或后续单独的订单诊断模块处理。012 Account Overview / One-Click Sync 不负责判断该订单是否成交。

011 只保存事实：

```text
本次成交明细查询成功，但 Binance 返回 0 条成交。
```

## 16. 查询失败处理

如果 011 查询 Binance 失败，例如：

```text
网络错误
超时
Binance 5xx
限频
连接异常
响应无法可信解析
```

则只能记录为：

```text
unknown
```

unknown 不代表：

```text
没有成交
订单失败
可以重新下单
可以释放订单锁
可以更新持仓
```

011 不做激进连续重试，不做自动重下单。

## 17. 异常恢复补同步

011 需要支持异常恢复补同步。

场景：

```text
010 已经查询到订单状态；
系统在 011 查询成交明细前崩溃或中断；
系统恢复后，需要补查这笔订单的成交明细。
```

恢复入口只能扫描：

```text
1. 已经有 010 成功订单状态证据。
2. 还没有 011 成交明细同步记录，或同步结果为 unknown。
3. 属于当前 active 市场域。
4. 仍在自动恢复窗口内。
```

P0 自动恢复窗口建议为：

```text
24 小时
```

如果超过自动恢复窗口：

```text
011 不自动请求 Binance；
记录 recovery_skipped_out_of_window；
交给人工排查或后续单独的订单诊断模块处理。
```

注意：

```text
012 Account Overview / One-Click Sync 不负责处理 011 超过恢复窗口的成交同步问题。
```

恢复补同步不得：

```text
1. 查询数据库历史最新订单。
2. 扫描全历史订单。
3. 查询超过恢复窗口的旧订单。
4. 更新持仓。
5. 释放订单锁。
6. 对账。
7. 撤单。
```

## 18. Alert / 审计规则

011 必须写审计。

建议：

```text
synced：
  普通审计，记录同步成交条数和汇总结果。

synced_empty：
  告警或高优先级审计。
  当前 009 是市价单，0 条成交不应被静默忽略。

unknown：
  告警。

failed_before_query：
  告警。

blocked_before_query：
  告警。

skipped_no_order_status：
  普通日志，不写风险告警。

recovery_skipped_out_of_window：
  告警或高优先级审计。
```

敏感信息不得落库：

```text
API secret
signature
完整认证头
```

011 可以保存 realized pnl 字段，因为 Binance 成交明细接口返回该字段；但 011 不得据此更新账户、持仓、绩效报表或释放订单锁。
