# 010 OrderStatusSync Requirements

> 建议路径：`docs/requirements/order_status_sync.md`  
> 阶段：010  
> 模块：OrderStatusSync / 订单状态同步  
> 上游：009 Order Submission  
> 下游：011 FillSync  
> 状态：需求定稿草案  
> 核心结论：010 只查询订单状态；011 才查询成交明细。

---

## 1. 阶段目的

010 只负责一件事：

```text
对本轮 009 实际提交过的市价单，重新向 Binance 查询订单状态，并保存查询结果作为审计证据。
```

010 不依赖 009 下单接口返回作为最终状态依据。

009 下单接口返回可以作为本地线索，但 010 的最终状态证据必须来自 010 自己重新请求 Binance 得到的数据。

---

## 2. 当前业务前提

当前 009 阶段提交的是：

```text
市价单
```

因此 010 不按长期挂单追踪设计。

010 不处理：

```text
长期挂单
订单过期
30 分钟自动撤单
GTD / goodTillDate
主动撤单
自动撤单
```

如果后续要做订单过期或撤单，应放到后续 CancelOrder / EmergencyCancel 阶段，不进入 010。

---

## 3. 010 和 011 必须分开

010 和 011 使用不同 Binance 接口，职责不能合并。

010 查询的是订单状态：

```text
USDS-M：GET /fapi/v1/order
COIN-M：GET /dapi/v1/order
```

011 查询的是成交明细：

```text
USDS-M：GET /fapi/v1/userTrades
COIN-M：GET /dapi/v1/userTrades
```

因此：

```text
010 只问：这笔订单 Binance 是否认识？当前订单状态是什么？
011 才问：这笔订单具体成交了几笔？每笔价格、数量、手续费是多少？
```

---

## 4. 010 的输入范围

正常情况下，010 只查询：

```text
本轮交易流程中，009 实际提交过的最近一笔订单。
```

这里的“最近一笔”必须理解为：

```text
本轮流程里的最近一次 009 提交。
```

不能理解为：

```text
数据库全局历史最新订单。
```

010 不得自己随意回查历史订单。

### 4.1 009 提交记录主线与外键要求

010 的订单状态查询记录必须以 009 提交记录作为主线关联对象。

业务要求：

```text
1. 009 提交记录是订单提交、状态查询、成交同步、对账链路的主入口。
2. 010 生成的每一条订单状态查询记录，都必须能直接关联回对应的 009 提交记录。
3. 未来 UI 打开某一条 009 提交记录详情时，必须能展开查看该订单对应的 010 状态查询结果。
4. 010 不得只按 Binance 订单号、客户订单编号、交易对或时间孤立保存查询结果。
5. 010 查询结果必须保留与 009 提交记录的直接可追溯关系。
```

允许关系：

```text
一条 009 提交记录可以对应 0 条、1 条或多条 010 状态查询记录。
```

含义：

```text
0 条：
  本轮 009 后 010 尚未执行，或本轮无 010 查询结果。

1 条：
  正常完成一次订单状态查询。

多条：
  查询失败后补查，或异常恢复后再次查询。
```

最终 UI / 审计主线应为：

```text
009 提交记录
  ├── 010 订单状态查询记录
  ├── 011 成交明细记录
  └── 011 订单成交汇总
```

012 Account Overview / One-Click Sync 是独立账户总览模块，不挂在 009 订单详情主线下，不作为订单对账结果。

这条是 P0 追溯要求。否则后续 UI、审计、排错会依赖 symbol、orderId、时间等字段拼接数据，风险较高。

---

## 5. 本轮无提交时的处理

如果本轮策略决定不操作，或者本轮没有发生 009 真实提交，则 010 必须跳过。

要求：

```text
1. 不请求 Binance。
2. 不回查历史最近订单。
3. 不生成订单状态查询快照。
4. 只记录普通运行日志，说明本轮没有订单需要查询。
5. 不写风险告警。
```

示例：

```text
4:00 策略决定买入，009 提交市价单 A，010 查询订单 A。

8:00 策略决定不操作，009 没有提交订单。
8:10 010 不得查询订单 A。
8:10 010 直接跳过，不请求 Binance。
```

---

## 6. 异常恢复补查

010 需要支持异常恢复补查。

场景：

```text
009 已经提交订单并落库；
系统在 010 查询前崩溃或中断；
系统恢复后，需要补查这笔漏掉的订单状态。
```

这类补查不是扫描历史订单，而是只补查：

```text
已经确认由 009 提交过；
但还没有 010 查询记录；
并且仍在自动恢复窗口内的订单。
```

P0 自动恢复窗口建议为：

```text
24 小时
```

例如：

```text
0:00 009 提交订单。
0:01 系统崩溃，010 没跑。
12:00 系统恢复。
这笔订单仍在 24 小时内，010 可以补查。
```

如果超过自动恢复窗口：

```text
010 不自动请求 Binance；
记录 recovery_skipped_out_of_window；
交给人工排查或后续单独的订单诊断模块处理。
```

注意：

```text
012 Account Overview / One-Click Sync 不负责处理 010 超过恢复窗口的订单状态问题。
```

恢复补查不得：

```text
1. 按数据库历史最新订单粗暴查询。
2. 扫描全历史订单。
3. 查询昨天、前天或更早的非恢复范围订单。
4. 做成交同步。
5. 做撤单。
6. 做持仓更新。
7. 做锁释放。
```

---

## 7. 查询编号规则

010 查询 Binance 时，优先使用：

```text
本系统在 009 下单时传给 Binance 的客户订单编号。
```

原因：

```text
如果 009 请求超时，本地可能拿不到 Binance 返回的交易所订单号；
但本系统自己生成并提交给 Binance 的客户订单编号仍可用于查询。
```

如果客户订单编号不可用，但本地有 Binance 返回的交易所订单号，可以使用交易所订单号查询。

如果两者都没有：

```text
010 不请求 Binance；
记录查询前失败；
写明缺少可查询订单编号；
写审计/告警。
```

---

## 8. 市场域规则

010 必须支持：

```text
USDS-M Futures / fapi
COIN-M Futures / dapi
```

但运行时只能激活一个市场域。

要求：

```text
1. 当前激活 U 本位时，只能查询 U 本位订单。
2. 当前激活币本位时，只能查询币本位订单。
3. 订单所属市场域与当前激活市场域不一致时，不得请求 Binance。
4. 不一致时必须记录阻断原因，并写审计/告警。
```

010 不能把 U 本位和币本位并行查询。

---

## 9. 查询结果分类

010 本地查询结果建议分为：

```text
synced
not_found
unknown
failed_before_query
blocked_before_query
skipped_no_submission
recovery_skipped_out_of_window
```

含义：

```text
synced：
  成功请求 Binance，并取得该订单状态。

not_found：
  Binance 明确回复该单笔订单未找到。

unknown：
  本次查询没有得到可信结论。
  例如网络错误、超时、Binance 5xx、限频、连接异常。

failed_before_query：
  查询前本地条件不满足。
  例如缺少可查询订单编号。

blocked_before_query：
  查询前被系统规则阻断。
  例如市场域不一致。

skipped_no_submission：
  本轮没有 009 提交记录，010 正常跳过。

recovery_skipped_out_of_window：
  异常恢复时发现漏查订单，但已经超过自动恢复窗口，010 不自动请求 Binance。
```

---

## 10. not_found 的严格定义

not_found 只能表示：

```text
010 针对本轮这笔订单向 Binance 发起单笔订单查询，
Binance 明确回复未找到该订单。
```

以下情况不能记为 not_found：

```text
1. 当前挂单列表里没看到。
2. 查询超时。
3. 网络错误。
4. Binance 5xx。
5. 限频。
6. 本地缺少订单编号。
```

not_found 不代表：

```text
可以重新下单
可以释放订单锁
可以更新持仓
可以认为风险解除
```

---

## 11. Binance 原始状态保存

010 可以保存 Binance 返回的原始订单状态，例如：

```text
NEW
PARTIALLY_FILLED
FILLED
CANCELED
EXPIRED
REJECTED
EXPIRED_IN_MATCH
```

但 010 不得解释这些状态并产生交易后动作。

特别是：

```text
即使 Binance 返回 FILLED，
010 也只保存“订单状态为 FILLED”，
不得处理成交明细、成交均价、手续费、持仓变化或订单锁释放。
```

---

## 12. 010 不处理成交明细

一笔 009 市价单可能被 Binance 拆成多笔成交，这是正常现象。

但该问题属于 011 FillSync。

010 不得处理：

```text
1. Binance 具体分成多少笔成交。
2. 每笔成交价格。
3. 每笔成交数量。
4. 总成交均价。
5. 手续费。
6. 手续费资产。
7. realized pnl。
8. 成交明细去重。
9. 成交明细汇总。
```

010 只保存订单状态查询结果。

---

## 13. 查询失败处理

如果 010 查询 Binance 失败，例如：

```text
网络错误
超时
Binance 5xx
限频
连接异常
```

则只能记录为：

```text
unknown
```

unknown 不代表：

```text
订单失败
订单不存在
可以重新下单
可以释放订单锁
可以更新持仓
```

010 不做激进连续重试，不做自动重下单。

---

## 14. 审计与日志

010 每次实际请求 Binance，都必须保存可追溯证据。

至少记录：

```text
1. 对应的 009 提交记录。
2. 查询对象。
3. 查询时间。
4. 当前激活市场域。
5. 请求摘要。
6. Binance 响应摘要。
7. Binance 原始订单状态。
8. 本地查询结果分类。
9. HTTP 状态。
10. Binance 错误码和错误信息。
11. trace 信息。
```

敏感信息不得落库：

```text
API secret
signature
完整认证头
```

本轮无提交时，可以只写普通运行日志，不生成订单状态快照。

## 15. Alert / 告警规则

建议：

```text
synced：
  普通审计。

not_found：
  告警或高优先级审计。

unknown：
  告警。

failed_before_query：
  告警。

blocked_before_query：
  告警。

skipped_no_submission：
  普通日志，不写风险告警。

recovery_skipped_out_of_window：
  告警或高优先级审计，提示需要后续对账或人工处理。
```

## 17. 测试要求

### 17.1 本轮有 009 提交，查询成功

期望：

```text
1. 010 请求 Binance 订单查询接口。
2. 保存订单状态查询结果。
3. 本地结果为 synced。
4. 不处理成交明细。
5. 不更新持仓。
6. 不释放订单锁。
```

### 17.2 本轮无 009 提交

期望：

```text
1. 不请求 Binance。
2. 不回查历史最近订单。
3. 不生成订单状态查询快照。
4. 只记录普通运行日志。
5. 不写风险告警。
```

### 17.3 数据库存在历史订单，但本轮无提交

期望：

```text
1. 010 不查询历史订单。
2. 010 不请求 Binance。
3. 010 正常跳过。
```

### 17.4 009 提交结果 unknown

期望：

```text
1. 010 使用本轮订单编号查询 Binance。
2. 查到订单则记录 synced。
3. 明确未找到则记录 not_found。
4. 查询失败则记录 unknown。
5. 不重新下单。
6. 不释放订单锁。
```

### 17.5 0:00 提交后系统崩溃，12:00 恢复

期望：

```text
1. 010 识别这属于异常恢复补查。
2. 该订单在 24 小时恢复窗口内。
3. 010 可以请求 Binance 查询。
4. 查询结果正常落库。
5. 不处理成交明细。
```

### 17.6 超过恢复窗口

期望：

```text
1. 010 不自动请求 Binance。
2. 记录 recovery_skipped_out_of_window。
3. 写告警或高优先级审计。
4. 交给人工排查或后续单独诊断模块处理。
```

### 17.7 市场域不一致

期望：

```text
1. 不请求 Binance。
2. 记录 blocked_before_query。
3. 写审计/告警。
```

### 17.8 缺少可查询订单编号

期望：

```text
1. 不请求 Binance。
2. 记录 failed_before_query。
3. 写审计/告警。
```

### 17.9 查询失败

期望：

```text
1. 网络错误、超时、5xx、限频记录为 unknown。
2. 不记录 not_found。
3. 不重新下单。
4. 不释放订单锁。
5. 不更新持仓。
```

### 17.10 Binance 明确未找到订单

期望：

```text
1. 记录 not_found。
2. 保存 Binance 错误信息。
3. 不重新下单。
4. 不释放订单锁。
5. 不更新持仓。
```

### 17.11 Binance 返回 FILLED

期望：

```text
1. 保存原始订单状态 FILLED。
2. 本地结果为 synced。
3. 不查询成交明细。
4. 不计算成交均价。
5. 不计算手续费。
6. 不更新持仓。
7. 不释放订单锁。
```

### 17.12 009 提交记录主线关联

期望：

```text
1. 010 订单状态查询记录必须直接关联对应的 009 提交记录。
2. 打开某条 009 提交记录时，可以找到该订单对应的 010 查询结果。
3. 同一条 009 提交记录允许对应多条 010 查询记录。
4. 不允许只靠 symbol、Binance orderId、客户订单编号或时间范围反查拼接。
```

### 17.13 U 本位与币本位

期望：

```text
1. U 本位走 /fapi/v1/order。
2. 币本位走 /dapi/v1/order。
3. 不混用市场域。
4. 币本位不转换为 base asset 或 quote asset。
```

## 18. 与后续阶段关系

010 完成后，订单详情主线继续：

```text
011 FillSync：
  查询并保存成交明细，并生成以 009 提交记录为主线的订单成交汇总。
```

012 是独立账户总览模块：

```text
012 Account Overview / One-Click Sync：
  基于 006 账户同步能力，一键同步和展示 Binance 原始账户 / 持仓快照。
  不绑定 009 / 010 / 011。
  不做订单对账。
  不释放订单锁。
  不阻断交易。
```

如果后续需要订单诊断或订单对账，应单独设计诊断模块，不属于 010 / 011 / 012 当前范围。

010 不得提前实现这些内容。

---

## 19. 最终结论

010 的职责是：

```text
只查询本轮 009 实际提交过的市价单订单状态；
使用 010 自己重新请求 Binance 得到的数据作为状态证据；
保存查询结果；
不处理成交、不撤单、不释放锁、不更新持仓、不回查历史订单。
```

如果本轮没有 009 提交：

```text
010 直接跳过，不请求 Binance，不回查历史最近订单。
```

如果 009 已提交但 010 前系统中断：

```text
010 允许在 24 小时自动恢复窗口内补查漏掉的订单状态。
```
